What is GenLayer

A blockchain where smart contracts
can think, search, and decide

GenLayer is a new kind of blockchain where validator nodes, each powered by a different AI model, reach consensus on decisions that require reasoning — not just arithmetic. It closes the gap between what smart contracts can do and what real-world problems actually need.

Web access

Live internet access — no oracle

Traditional smart contracts are blind to the outside world. GenLayer contracts fetch any public URL directly at runtime. No Chainlink feed, no relay contract, no trust assumption on a data provider.

gl.nondet.web.render(url, mode="text")
AI reasoning

LLM calls inside consensus

Your contract can call a language model as part of its logic. The result isn't just one model's opinion — multiple validators each call their own LLM and the network agrees through Optimistic Democracy.

gl.nondet.exec_prompt(text)
Python

Write contracts in Python

No need to learn Solidity or Rust. Intelligent Contracts are Python classes. You get the full standard library, dataclasses, type hints, JSON, and the entire GenLayer SDK through a single import.

from genlayer import *

How FreelanceGuard works

Five steps from job post
to automatic fund release

Every step happens on-chain. No backend server, no database, no human moderator. The contract itself fetches evidence, evaluates it with AI, and moves money.

01

Client posts a job and locks GEN in escrow

The client calls post_job() with the freelancer's wallet address, a title, and plain-English acceptance criteria — the exact description of what "done" means. The GEN token budget is sent with the transaction via gl.message.value and locked inside the contract. No third-party escrow service, no platform custody.

💡

The criteria field is just plain text — e.g. "A deployed React app with a hero section, feature list, and working contact form." No encoding. The LLM reads it directly.

@gl.public.write.payable · gl.message.value
02

Freelancer submits their work as a public URL

When the work is done, the freelancer calls submit_work() with a URL pointing to the delivered work — a deployed website, a GitHub repo, a Loom video, a Google Doc, or any publicly accessible resource. The contract verifies that only the assigned freelancer can submit, using gl.message.sender_account.

@gl.public.write · gl.message.sender_account
03

Contract fetches the submission URL live

Anyone can call evaluate_submission() to trigger the AI evaluation. The contract calls gl.nondet.web.render(url, mode="text") to fetch the actual live content of the submitted URL. Each validator fetches independently — there is no shared fetch, no relay, no oracle.

🌐

No oracle needed. Traditional blockchains need an oracle to bring external data on-chain. GenLayer contracts access the web natively — the fetch happens inside consensus, not outside it.

gl.nondet.web.render(url, mode="text")
04

Multiple AI validators evaluate the work independently

The fetched content and the original criteria are fed into an LLM prompt via gl.nondet.exec_prompt(). The leader validator proposes an "accepted" or "rejected" verdict. Other validators independently re-run the same contract with their own LLM (GPT-4, Claude, LLaMA, Gemini, etc.). A secondary LLM call implements the Equivalence Principle — asking whether the two verdicts agree semantically.

⚖️

Why semantic comparison instead of string equality? GPT-4 might say "the work is accepted" while Claude says "deliverables confirmed." Both mean the same thing — string comparison would wrongly fail. The Equivalence Principle fixes this.

gl.vm.run_nondet_unsafe · Equivalence Principle
05

Funds released automatically based on consensus verdict

Once Optimistic Democracy reaches a supermajority consensus, gl.emit_transfer() executes. If the verdict is "accepted", the escrowed GEN goes to the freelancer. If "rejected", it is refunded to the client. The entire flow — from web fetch to fund release — is on-chain, auditable, and irreversible.

gl.emit_transfer(address, amount)

How Optimistic Democracy works — step by step

Optimistic Democracy is GenLayer's core consensus mechanism. It assumes the leader is correct ("optimistic") and lets validators challenge that assumption. Here's the exact sequence for a FreelanceGuard evaluation:

Step 1

Leader executes contract

One validator is elected as leader. It runs evaluate_submission(), fetches the URL, calls the LLM, and proposes a verdict: accepted or rejected with a reason.

Step 2

Validators receive the proposal

All other active validators receive the leader's proposed result. Each independently re-runs the same contract logic with their own LLM provider. Results will differ in wording.

Step 3

Equivalence check per validator

Each validator calls a second LLM with both the leader's result and their own, asking: "Do these agree on accept/reject?" This is the Equivalence Principle — semantic, not syntactic comparison.

Step 4

Supermajority vote

Validators submit agree/disagree votes. If a supermajority agrees with the leader's verdict, the transaction is accepted and finalized. If not, a new leader is elected and the round repeats.

🔐

Slashing for dishonesty. Validators stake GEN tokens. If a validator consistently disagrees with consensus without good reason, they lose stake. This creates a financial incentive to run evaluations honestly, which secures the whole system.

# { "Depends": "py-genlayer:1jb45aa8ynh2a9c9xn3b7qqh8sm5q93hwfp7jqmwsfhh8jpz09h6" }
# ↑ Required header. Declares the GenVM SDK version. Always include this.

from genlayer import *
# ↑ Imports gl (the GenLayer SDK), TreeMap, DynArray, u256, and more.
#   This single import gives you everything you need.

from dataclasses import dataclass
import json
# ↑ Standard Python library — works as expected inside GenVM.

@dataclass
class Job:
    job_id:          int
    client:          str    # wallet address of the client
    freelancer:      str    # wallet address of the freelancer
    title:           str
    criteria:        str    # plain-English acceptance criteria — read by LLM directly
    budget:          u256   # u256 = GenLayer's native 256-bit uint for token amounts
    submission_url:  str    # URL to the delivered work — will be fetched live
    verdict:         str    # "pending" | "accepted" | "rejected"
    verdict_reason:  str    # LLM's explanation — stored on-chain
    is_funded:       bool
    is_submitted:    bool

class FreelanceGuard(gl.Contract):
    # ── PERSISTENT STATE ───────────────────────────────────────────────────────
    # Use TreeMap[K, V] instead of dict — regular Python dicts are NOT stored
    # on-chain between transactions. This is the most common beginner mistake.
    # Use DynArray[T] instead of list for the same reason.

    jobs:            TreeMap[int, Job]
    next_job_id:     int
    client_jobs:     TreeMap[str, DynArray[int]]
    freelancer_jobs: TreeMap[str, DynArray[int]]

    def __init__(self):
        self.next_job_id = 0
        self.jobs = TreeMap()
        self.client_jobs = TreeMap()
        self.freelancer_jobs = TreeMap()
    @gl.public.write.payable
    # ↑ This decorator allows the method to RECEIVE GEN tokens.
    #   Without .payable, the transaction will fail if value is sent.
    #   The tokens are held by the contract (escrow) until released.
    def post_job(
        self,
        freelancer_address: str,
        title: str,
        criteria: str,
    ) -> int:
        # gl.message.value = the GEN tokens sent with this transaction
        # gl.message.sender_account = the wallet that called this method
        assert gl.message.value > 0, "Budget must be greater than 0 GEN"
        assert len(criteria) >= 20, "Criteria must be descriptive"

        job_id = self.next_job_id
        self.next_job_id += 1

        job = Job(
            job_id=job_id,
            client=gl.message.sender_account, # who called this method
            freelancer=freelancer_address,
            title=title,
            criteria=criteria,    # stored as-is — plain English!
            budget=gl.message.value,          # GEN locked in escrow
            submission_url="",
            verdict="pending",
            verdict_reason="",
            is_funded=True,
            is_submitted=False,
        )
        self.jobs[job_id] = job

        # Track which jobs belong to each address
        if gl.message.sender_account not in self.client_jobs:
            self.client_jobs[gl.message.sender_account] = DynArray()
        self.client_jobs[gl.message.sender_account].append(job_id)

        return job_id  # caller gets back the job ID for future calls
    @gl.public.write
    # ↑ @gl.public.write = anyone can call this (it's public),
    #   and it can modify contract state (write = state-changing).
    #   The assert below restricts WHO can actually use it.
    def submit_work(self, job_id: int, submission_url: str) -> None:
        assert job_id in self.jobs, "Job does not exist"
        job = self.jobs[job_id]

        # gl.message.sender_account is the caller's wallet address.
        # This is the on-chain equivalent of msg.sender in Solidity.
        # It's cryptographically verified — cannot be spoofed.
        assert gl.message.sender_account == job.freelancer, \
            "Only the assigned freelancer can submit"

        assert job.verdict == "pending", "Job is already resolved"
        assert job.is_funded, "Job has not been funded"
        assert submission_url.startswith("http"), "Must be a valid URL"

        job.submission_url = submission_url
        job.is_submitted = True
        self.jobs[job_id] = job
        # ↑ Must reassign the whole Job back to TreeMap.
        #   Mutating job in place is NOT enough — you must write it back.
    @gl.public.write
    def evaluate_submission(self, job_id: int) -> None:
        job = self.jobs[job_id]
        assert job.is_submitted, "No submission yet"
        assert job.verdict == "pending", "Already resolved"

        # ── STEP 1: Fetch the live submission ──────────────────────────────────
        # gl.nondet.web.render() fetches any public URL at runtime.
        # "nondet" = non-deterministic (different validators may get slightly
        # different content, but the verdict comparison handles this).
        # Each validator fetches independently — no shared oracle, no relay.
        try:
            content = gl.nondet.web.render(job.submission_url, mode="text")
            content = content[:3000]  # trim to avoid LLM token limit
        except:
            content = f"[Could not fetch: {job.submission_url}]"

        # ── STEP 2: Build the evaluation prompt ────────────────────────────────
        # The prompt passes the original criteria (plain English) and the
        # fetched content. The LLM evaluates one against the other.
        prompt = f"""
You are an impartial freelance work evaluator on a blockchain platform.

JOB TITLE: {job.title}
ACCEPTANCE CRITERIA: {job.criteria}
SUBMISSION URL: {job.submission_url}
FETCHED CONTENT: {content}

Evaluate whether the submission satisfies ALL criteria.
Respond ONLY as valid JSON, no extra text:
{{"verdict": "accepted" or "rejected",
  "reason": "one paragraph explanation"}}
"""

        # ── STEP 3: Run consensus (see next tab) ───────────────────────────────
        result = gl.vm.run_nondet_unsafe(
            lambda: gl.nondet.exec_prompt(prompt, response_format="json"),
            validator_fn
        )

        # ── STEP 4: Update state and release funds ─────────────────────────────
        job.verdict = result.get("verdict", "rejected")
        job.verdict_reason = result.get("reason", "")
        self.jobs[job_id] = job

        if job.verdict == "accepted":
            gl.emit_transfer(job.freelancer, job.budget)  # 🎉 pay freelancer
        else:
            gl.emit_transfer(job.client, job.budget)      # 🔄 refund client
        # ── CONSENSUS LOGIC ────────────────────────────────────────────────────
        #
        # gl.vm.run_nondet_unsafe(leader_fn, validator_fn) runs the
        # Optimistic Democracy consensus process:
        #
        #   leader_fn   → called on the LEADER validator. Returns the proposed result.
        #   validator_fn → called on every OTHER validator. Returns True = agree.
        #
        # The "unsafe" suffix means YOU are responsible for defining equivalence.
        # A safe version would use strict equality — useless for LLM outputs.

        def leader_fn():
            # Leader calls its LLM and proposes the verdict
            return gl.nondet.exec_prompt(prompt, response_format="json")

        def validator_fn(leaders_result) -> bool:
            # Validator re-runs the same evaluation with ITS OWN LLM
            if not isinstance(leaders_result, gl.vm.Return):
                return False

            my_result = gl.nondet.exec_prompt(prompt, response_format="json")

            # ── THE EQUIVALENCE PRINCIPLE ───────────────────────────────────
            # Instead of comparing strings (would fail — LLMs produce different
            # wording every time), we use a SECOND LLM call to check whether
            # both verdicts agree on the accept/reject decision.
            #
            # "The work is accepted" == "Deliverables confirmed" == True
            # "Work is rejected, missing form" == "Accepted" == False

            eq_prompt = f"""
Two AI evaluators independently assessed the same freelance submission.

Evaluator A verdict: {json.dumps(leaders_result.calldata)}
Evaluator B verdict: {json.dumps(my_result)}

Do both evaluators agree on whether the work should be ACCEPTED or REJECTED?
Respond ONLY as JSON: {{"agree": true}} or {{"agree": false}}
"""
            comparison = gl.nondet.exec_prompt(eq_prompt, response_format="json")
            return comparison.get("agree", False)

        result = gl.vm.run_nondet_unsafe(leader_fn, validator_fn)
        # ↑ Returns the leader's result if supermajority of validators agreed.
🧠

The key insight: The criteria field is plain English — no encoding, no hashing. The LLM reads it like a human moderator would, then evaluates the submission against it. This is what separates an Intelligent Contract from a traditional smart contract: business logic can be expressed in natural language.

Deploy in 30 minutes

GenLayer Studio is a fully browser-based IDE. No local setup, no wallet extension, no faucet. Open the URL and start building.

01

Open GenLayer Studio

Navigate to studio.genlayer.com. The Studio opens with a test wallet already loaded with testnet GEN tokens. You'll see a file explorer on the left, code editor in the center, and transaction panel on the right.

02

Create the contract file

Click the + icon next to "Contracts" in the left panel. Name the file freelance_guard.py. Delete the placeholder code. Paste the full contract from the Code section above (use the "Setup & state" tab as your starting point).

03

Deploy to testnet

Click Deploy in the right panel. The constructor takes no parameters — just click Confirm. Wait a few seconds for the transaction to finalize. Your contract address will appear — save it.

04

Post a test job as client

In "Write Methods", expand post_job. Fill in a freelancer address (use your second test account), a title, and detailed criteria. Set value to 10 GEN. Click Run. You'll get back job_id: 0.

05

Submit work as freelancer

Switch to the second test account in Studio. Call submit_work with job_id: 0 and a real public URL — a deployed site, a GitHub README, a Loom video. The URL must be accessible without login.

06

Trigger evaluation and watch consensus

Call evaluate_submission(0). Watch the transaction status move through: Pending → Proposing → Committing → Revealing → Finalized. Click the transaction hash to see each validator's verdict and the equivalence check results.

07

Read the verdict on-chain

Call get_job(0) in "Read Methods". Inspect the verdict and verdict_reason fields. If accepted, budget will be 0 — the GEN was transferred to the freelancer. If rejected, it was refunded to the client.

Key concepts

GenLayer concepts you
just used

Every feature in FreelanceGuard maps directly to a core GenLayer primitive. Understanding these lets you build any Intelligent Contract.

Intelligent Contract
A Python class extending gl.Contract. Unlike traditional smart contracts, it can access the web, call LLMs, and reason about unstructured data — all inside the on-chain consensus process. Every method you define becomes a callable on-chain function.
gl.nondet.web.render()
Fetches any public URL at the time the transaction executes. Returns the page content as text or HTML. Each validator fetches independently. This replaces the need for oracle contracts entirely — web access is native to GenLayer. The "nondet" prefix means the result may vary slightly between validators, which is handled by the Equivalence Principle.
gl.nondet.exec_prompt()
Calls an LLM with a text prompt. Each validator uses their own configured LLM provider (GPT-4, Claude, LLaMA, etc.). Returns structured output when response_format="json" is specified. The non-determinism is resolved by the Equivalence Principle — a second LLM call that compares semantic meaning rather than exact text.
Optimistic Democracy
GenLayer's consensus mechanism. A leader validator proposes a result (optimistic assumption: the leader is honest). Other validators independently verify using validator_fn. If a supermajority agrees, the result is finalized. If they disagree, a new leader is elected and the round repeats. Validators stake GEN and are slashed for dishonest behavior.
Equivalence Principle
The rule for deciding when two validators' LLM outputs "count" as the same. Instead of string equality (which fails because LLMs produce different wording), GenLayer uses a second LLM call to check semantic equivalence. "Accepted — all criteria met" and "Work confirmed, deliverables present" are semantically equivalent even though they're different strings.
gl.emit_transfer()
Sends GEN tokens from the contract's balance to any address. Used in FreelanceGuard to automatically release escrowed funds — to the freelancer on acceptance, or back to the client on rejection. This is the "automated execution" part of the trustless escrow: no human needs to approve the payment.
gl.message.sender_account
The wallet address of whoever called the current method. Cryptographically verified — cannot be spoofed. Used in FreelanceGuard to ensure only the assigned freelancer can submit work. Equivalent to msg.sender in Solidity.
TreeMap / DynArray
GenLayer's persistent on-chain storage types. Always use TreeMap[K, V] instead of Python dict and DynArray[T] instead of Python list for any data that must survive between transactions. Regular Python dicts and lists exist only in memory during execution and are lost immediately after.
@gl.public.write.payable
A method decorator. @gl.public makes it callable from outside. .write allows state modification. .payable allows GEN tokens to be sent with the call. Without .payable, sending tokens with a transaction causes it to revert. Use @gl.public.view for read-only methods that cannot modify state.
gl.vm.run_nondet_unsafe()
Executes the Optimistic Democracy consensus round. Takes two functions: leader_fn() which runs on the leader validator and returns the proposed result, and validator_fn(leaders_result) which runs on every other validator and returns True if they agree. The "unsafe" in the name means you define equivalence yourself — you're responsible for the comparison logic.
